# US212 - Create Scheduled Flight

## 1. User Story
> As a Backoffice Operator, I want to create a scheduled flight for an active route, assigning a specific aircraft and defining the departure date and time.

## 2. Acceptance Criteria
- The request must provide `routeId`, `aircraftRegistration`, and `departureTime` (ISO date-time format).
- **Domain Constraints & Validations:**
    - The assigned `Route` must exist and be currently active.
    - The assigned `Aircraft` must exist, be operational, and cannot be undergoing maintenance during the flight's timeframe.
    - The aircraft's maximum range must be greater than or equal to the route's distance requirement.
    - The aircraft cannot have overlapping flight schedules (scheduling conflict checks).
- On success, the system returns HTTP `201 Created` with the flight payload and HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The referenced route and aircraft resources must already be registered in the persistent database layer.

## 4. Post-conditions
- The `ScheduledFlight` instance is successfully saved to the database.
- The aircraft availability state window shifts to blocked for the designated timeframe.

## 5. Main Success Scenario
1. The actor sends a `POST /api/scheduled-flights` request containing the valid configuration JSON payload.
2. The `ScheduledFlightController` interceptor validates the payload parameters syntax.
3. The `CreateScheduledFlightUseCase` orchestrates validations across `RouteRepository`, `AircraftRepository`, and `MaintenanceRepository`.
4. Domain checks pass (range is sufficient, no schedule overlap, no maintenance block).
5. The `ScheduledFlight` entity is instantiated and committed via the `ScheduledFlightRepository`.
6. The system answers with an HTTP `201 Created` payload response.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user context is missing or lacks authorization clearances | HTTP `401 Unauthorized` or `403 Forbidden` |
| 3 | The designated `routeId` or `aircraftRegistration` is missing | HTTP `404 Not Found` |
| 4 | The aircraft has insufficient operational range for the selected route | HTTP `400 Bad Request` |
| 4 | The aircraft has a scheduling time conflict or is in maintenance | HTTP `409 Conflict` (throws domain overlap error) |

## 7. Design Justification
- **Business Rule Encapsulation:** Multi-entity constraints (Route + Aircraft + Maintenance) require aggregate orchestration handled in the use case tier to isolate pure business coordination logic away from infrastructure.
