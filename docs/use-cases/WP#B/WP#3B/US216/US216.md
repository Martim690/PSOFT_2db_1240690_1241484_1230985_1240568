# US216 - Search Alternative Route Links

## 1. User Story
> As a Backoffice Operator, I want to query alternative connections between an origin and destination airport, evaluating both direct and one-stop paths to handle contingency flight planning.

## 2. Acceptance Criteria
- The endpoint requires `origin` and `destination` IATA airport string codes.
- The analysis must split results into two explicit target payload structures:
    - `directRoutes`: Direct path records matching the criteria.
    - `oneStopRoutes`: Valid multi-leg connection paths routing through an intermediate hub airport.
- On success, returns HTTP `200 OK` containing both lists.

## 3. Pre-conditions
- The user must have active system validation clearances.

## 4. Post-conditions
- Read-only lookup.

## 5. Main Success Scenario
1. The actor sends a `GET /api/routes/alternatives?origin=LIS&destination=OPO` request.
2. The controller forwards query strings to the graph navigation algorithm in the use case layer.
3. The algorithm runs depth checks against the `RouteRepository` data.
4. Path groupings map safely into a structured combined JSON response and return an HTTP `200 OK`.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | Authorization credentials missing | HTTP `401 Unauthorized` |
| 2 | IATA parameters provided fail syntactical boundaries (e.g., invalid length) | HTTP `400 Bad Request` |

## 7. Design Justification
- **Algorithmic Path Separation:** Decoupling the graph query paths into direct vs one-stop models optimizes response parsing for edge-case scheduling tasks without heavy computational complexity.
