# US204 - View Top 5 Most Utilized Aircraft Models

## 1. User Story
> As a Backoffice Operator, I want to see the Top 5 most utilized aircraft models based on total flight hours or number of assignments.

## 2. Acceptance Criteria
- The request can include an optional query parameter `sortBy` (valid values: `hours`, `assignments`). If omitted or both selected, it defaults to `hours`.
- The system must aggregate data from all physical aircraft instances, grouping them by their respective `AircraftModel`.
- The returned list must contain a maximum of 5 entries, sorted in descending order according to the requested criteria.
- The payload for each entry must include: `modelName`, `manufacturer`, `totalFlightHours`, and `totalAssignments`.
- On success, the system must return HTTP `200 OK` with the collection of models and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- For meaningful results, aircraft models and registered aircraft instances should exist in the system.

## 4. Post-conditions
- The state of the database remains completely unchanged (this is a read-only reporting operation).
- The Backoffice Operator is presented with actionable metrics regarding fleet model utilization.

## 5. Main Success Scenario
1. The actor sends a `GET /api/aircraft-models/top-utilized?sortBy=hours` request.
2. The `AircraftModelController` receives the request and delegates the logic to the `GetTopUtilizedModelsUseCase`.
3. The Use Case retrieves all physical aircraft instances from the `AircraftRepository`.
4. The system groups the instances by their associated `AircraftModel`.
5. The system calculates the sum of `totalFlightHours` and `numberOfAssignments` for each model group.
6. The system sorts the aggregated list in descending order based on the `sortBy` parameter and limits the output to the top 5.
7. The system maps the final list to `TopUtilizedModelResponse` DTOs.
8. The system returns an HTTP `200 OK` response containing the top 5 collection and HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1    | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2    | There are no aircraft instances registered in the system | HTTP `200 OK` with an empty collection |
| 3    | There are fewer than 5 models in the entire fleet | HTTP `200 OK` returning all available models sorted correctly |

## 7. Design Justification
- **In-Memory Aggregation vs DB Query:** For the current scope, the grouping and summing logic is implemented via Java Streams in the Use Case (`aircraftRepository.findAll()`). This makes the business logic explicitly testable in Java. If the fleet scales to millions of aircraft, this logic should be refactored into an optimized SQL `@Query` with `GROUP BY` and `ORDER BY` directly in the Repository to avoid high memory consumption.
- **DTO Mapping:** `TopUtilizedModelResponse` prevents the exposure of domain entity relationships and exposes only the flat analytical data required by the frontend.
