# US201 - Update Aircraft Model Specifications

## 1. User Story
> As a Backoffice Operator, I want to update an aircraft model's specifications.

## 2. Acceptance Criteria
- The request must specify the target aircraft model via its `modelName` in the URL path.
- The request body may contain one or more optional fields: `fuelCapacity`, `maximumRange`, and `cruisingSpeed`.
- If a field is omitted (or sent as null) in the payload, the system must retain the existing value for that property.
- Numeric constraints apply: any updated `fuelCapacity`, `maximumRange`, or `cruisingSpeed` must be strictly positive integers (greater than 0).
- On success, the system must return HTTP `200 OK` with the updated model payload and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- An aircraft model with the specified `modelName` must already exist in the database.

## 4. Post-conditions
- The `AircraftModel` properties are successfully updated and persisted to the database.
- Future operations involving this aircraft model (like checking compatible routes) will now use the updated specifications.

## 5. Main Success Scenario
1. The actor sends a `PATCH /api/aircraft-models/{modelName}` request containing a JSON payload with the fields to be updated.
2. The `AircraftModelController` validates the syntax of the incoming request.
3. The `UpdateAircraftModelSpecsUseCase` retrieves the existing `AircraftModel` from the `AircraftModelRepository`.
4. The Use Case extracts the new values and calls the domain entity to update its specifications, ignoring any null values.
5. The `AircraftModel` entity enforces domain rules and applies the changes.
6. The `AircraftModelRepository` persists the updated entity to the database.
7. The system returns an HTTP `200 OK` response containing the updated model data and HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | The request payload violates numeric constraints (e.g., negative fuel capacity) | HTTP `400 Bad Request` (intercepted by `GlobalExceptionHandler`) |
| 3 | The specified `modelName` does not exist in the repository | HTTP `400 Bad Request` (throws `IllegalArgumentException` mapped to bad request/not found) |

## 7. Design Justification
- HTTP `PATCH` is used instead of `PUT` to allow partial updates, ensuring the client doesn't need to send the entire object to change just one field.
- The Domain Entity (`AircraftModel`) remains responsible for its own integrity validation during the update, preventing invalid states.
- The Use Case orchestrates the database fetch and the partial update logic, keeping the Controller lean and focused entirely on HTTP and HATEOAS representation.
