# US203 - View Compatible Routes for an Aircraft

## 1. User Story
> As an ATCC, I want to view which routes are compatible with a specific aircraft based on its range and capacity. The same airplane model can have different seat configuration, and therefore different capacities.

## 2. Acceptance Criteria
- The request must specify the target aircraft via its `registrationNumber` in the URL path.
- The system must return a list of all active routes that are compatible with the specified aircraft.
- A route is considered compatible if and only if:
    - The aircraft's `maximumRange` (from its model) is greater than or equal to the route's `distance`.
    - The aircraft's specific instance `seatingCapacity` is greater than or equal to the route's `minPassengerCapacity` requirement.
- The route must be currently active.
- On success, the system must return HTTP `200 OK` with the list of compatible routes and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `ATCC` or `ADMIN` role.
- An aircraft with the specified `registrationNumber` must exist in the database.
- Routes must exist in the system to be evaluated.

## 4. Post-conditions
- The state of the database remains unchanged (this is a read-only operation).
- The `ATCC` receives the necessary data to make assignment decisions.

## 5. Main Success Scenario
1. The actor sends a `GET /api/aircrafts/{registrationNumber}/compatible-routes` request.
2. The `AircraftController` handles the request and calls the `GetCompatibleRoutesUseCase`.
3. The Use Case retrieves the `Aircraft` entity (and its associated `AircraftModel`) using the `registrationNumber`.
4. The Use Case retrieves all available `Route` entities from the `RouteRepository`.
5. The system filters the routes by comparing the aircraft's physical constraints against the route's requirements.
6. The system maps the filtered routes to `CompatibleRouteResponse` DTOs.
7. The system returns an HTTP `200 OK` response containing the collection of compatible routes with HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1    | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2    | The specified `registrationNumber` does not exist in the repository | HTTP `404 Not Found` or `400 Bad Request` |
| 3    | There are no routes currently registered in the system | HTTP `200 OK` with an empty collection |
| 4    | No routes match the aircraft's constraints | HTTP `200 OK` with an empty collection |

## 7. Design Justification
- **Seat Configuration Isolation:** The evaluation uses the capacity from the `Aircraft` instance, not the `AircraftModel`, addressing the business rule that instances of the same model can have different configurations.
- **Performance:** For huge datasets, this logic could be pushed to a database query (e.g., via `@Query`), but doing it in memory via Streams keeps the domain logic explicitly visible in the Java code for maintainability.
