# US205 - View Real-Time Fleet Availability Status

## 1. User Story
> As an ATCC, I want to view real-time aircraft availability status (available, in-flight, under maintenance, inactive).

## 2. Acceptance Criteria
- The request must trigger the generation of a fleet status dashboard/report.
- The system must aggregate data from all physical aircraft instances registered in the fleet.
- The response must group aircraft by their current operational status (e.g., `AVAILABLE`, `IN_FLIGHT`, `UNDER_MAINTENANCE`, `INACTIVE`).
- The payload must contain the `totalAircraft` count in the fleet.
- The system must guarantee that all possible aircraft statuses are present in the response object, assigning a count of `0` to any status that currently has no aircraft.
- On success, the system must return HTTP `200 OK` with the fleet status payload and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `ATCC` or `ADMIN` role.
- (Optional but recommended) Aircraft instances should exist in the database to show non-zero metrics.

## 4. Post-conditions
- The state of the database remains completely unchanged (read-only operation).
- The ATCC is provided with real-time situational awareness of the fleet.

## 5. Main Success Scenario
1. The actor sends a `GET /api/aircrafts/fleet-status` request to view the dashboard.
2. The `AircraftController` handles the request and delegates the processing to the `GetFleetStatusUseCase`.
3. The Use Case retrieves the full list of `Aircraft` entities from the `AircraftRepository`.
4. The system groups the instances by their current status and counts the occurrences for each group.
5. The system iterates over all predefined `AircraftStatus` enum values to populate any missing statuses with a `0` count.
6. The system encapsulates the results in a `FleetStatusResponse` DTO.
7. The system returns an HTTP `200 OK` response containing the aggregated data and HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1    | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2    | There are no aircraft instances currently registered in the system | HTTP `200 OK` with `totalAircraft` as `0` and all statuses showing a count of `0` |

## 7. Design Justification
- **Consistent UI Dashboard:** By enforcing that the backend populates all possible statuses (even those with a `0` count), we drastically reduce the complexity on the frontend/client side, ensuring the UI always has the same keys to map in its charts/graphs.
- **DTO Encapsulation:** The Use Case maps the entity counts into a specific `FleetStatusResponse` record, preventing domain leakage and avoiding the exposure of internal data models via HTTP.
