# US205 - Get Real-Time Fleet Status

## 1. User Story
> As an ATCC, I want to view real-time aircraft availability status (available, in-flight, under maintenance, inactive).

## 2. Acceptance Criteria
- The request must be a `GET` to `/api/aircrafts/fleet-status`.
- The endpoint must be restricted to users with `ATCC` or `ADMIN` roles.
- The system must return a JSON response containing two main sections:
    - `statusCounts`: An aggregated map showing the total number of aircraft in each state (e.g., `AVAILABLE`: 2, `IN_FLIGHT`: 1).
    - `aircrafts`: A detailed list of objects, each containing the `registrationNumber` and the current `status` of an aircraft.
- The calculation must be processed efficiently in memory to minimize database hits.
- On success, the system returns HTTP `200 OK` with the constructed payload.

## 3. Pre-conditions
- The actor must be authenticated and have the `ATCC` or `ADMIN` role.
- Aircraft instances must exist in the database for the response to be populated (otherwise it returns empty counts/lists, which is still a valid 200 OK).

## 4. Post-conditions
- The system simply queries and returns the current operational state without modifying any database records.

## 5. Main Success Scenario
1. The actor sends a `GET /api/aircrafts/fleet-status` request.
2. The `AircraftController` receives the request and calls the `GetFleetStatusUseCase`.
3. The Use Case requests all aircraft instances from the `AircraftRepository` via `findAll()`.
4. The Use Case uses Java Streams to group and count the aircraft by their status (`statusCounts`).
5. The Use Case uses Java Streams to map the list of aircraft into lightweight DTOs (`aircrafts`).
6. The Use Case constructs a `FleetStatusResponse` encompassing both the counts and the details.
7. The system returns an HTTP `200 OK` with the JSON response.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated | HTTP `401 Unauthorized` |
| 1 | The user lacks the `ATCC` or `ADMIN` role (e.g., BACKOFFICE_OPERATOR) | HTTP `403 Forbidden` |
| 3 | There are no aircraft registered in the system | HTTP `200 OK` (Returns `{ "statusCounts": {}, "aircrafts": [] }`) |

## 7. Design Justification
- **Single Payload Pattern:** Returning both the summary (`counts`) and the specific details (`list`) in a single payload prevents the frontend from making multiple HTTP requests to build the dashboard, reducing network overhead and ensuring data consistency.
- **In-Memory Calculation:** Instead of executing multiple `COUNT` queries in the database, the system performs a single `findAll()` and processes the aggregations using Java Streams. This is highly performant for typical fleet sizes.
