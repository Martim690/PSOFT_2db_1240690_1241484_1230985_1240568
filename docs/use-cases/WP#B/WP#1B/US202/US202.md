# US202 - Register Aircraft Model with Optional Diagram

## 1. User Story
> As a Backoffice Operator, I want to register an aircraft model with an optional image or technical diagram.

## 2. Acceptance Criteria
- The request must provide `manufacturer`, `modelName`, `fuelCapacity`, `maximumRange`, `cruisingSpeed`, and an optional `technicalDiagramUrl`.
- If the `technicalDiagramUrl` is not provided (or is null), the system must default its value to `"empty"` or handle it gracefully without failing.
- `modelName` must be unique; attempting to register a duplicate model name must be rejected.
- Numeric constraints must be respected (e.g., fuel capacity must be strictly positive).
- On success, the system must return HTTP `201 Created` with the model payload (including the diagram URL) and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- An aircraft model with the specified `modelName` must not already exist in the database.

## 4. Post-conditions
- The `AircraftModel` is successfully saved to the database, containing the link to the image/diagram if provided.
- The model is available for fleet instances registration.

## 5. Main Success Scenario
1. The actor sends a `POST /api/aircraft-models` request containing a JSON payload with the model's technical specifications and the optional diagram URL.
2. The `AircraftModelController` validates the syntax of the incoming DTO.
3. The `RegisterAircraftModelUseCase` verifies via the `AircraftModelRepository` that the `modelName` does not already exist.
4. The `AircraftModel` entity is instantiated. If the diagram URL is missing, it assigns the default `"empty"` value.
5. The `AircraftModelRepository` persists the new entity to the database.
6. The system returns an HTTP `201 Created` response containing the saved model data and HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | The request payload violates numeric constraints | HTTP `400 Bad Request` |
| 3 | The `modelName` already exists in the repository | HTTP `400 Bad Request` (throws `IllegalArgumentException`) |

## 7. Design Justification
- The inclusion of the image/diagram is treated as a URL string rather than a binary BLOB in the database to keep the database lightweight and performant.
- The Use Case handles the business logic of defaulting the missing image to "empty", keeping the controller agnostic of domain-specific fallback rules.
