# US202 - Register Aircraft Model with Physical Diagram Upload

## 1. User Story
> As a Backoffice Operator, I want to register an aircraft model with an optional image or technical diagram.

## 2. Acceptance Criteria
- The request must be a `POST` to `/api/aircraft-models` using `multipart/form-data`.
- The payload must include `manufacturer`, `modelName`, `fuelCapacity`, `maximumRange`, `cruisingSpeed`, and an optional `file`.
- `modelName` must be unique; duplicate names must be rejected.
- Numeric constraints (positive values) must be enforced.
- The system must physically save the uploaded file to `src/main/resources/static/diagrams/` and sanitize the filename.
- On success, the system must return HTTP `201 Created` with the model payload, including the full public URL to access the diagram, and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The `static/diagrams/` directory must be configured as a publicly accessible resource on the server.

## 4. Post-conditions
- The `AircraftModel` is persisted in the database with the filename reference.
- The diagram file is physically stored on the disk, making it accessible via HTTP.

## 5. Main Success Scenario
1. The actor sends a `POST /api/aircraft-models` request as `multipart/form-data`, including metadata and the image file.
2. The `AircraftModelController` validates the DTO and processes the `MultipartFile`.
3. The `RegisterAircraftModelUseCase` verifies that the `modelName` is unique.
4. If a file is provided, the Use Case sanitizes the filename, copies the file to the `static/diagrams/` directory, and maps the filename to the entity. If no file is provided, it assigns the default value `"empty.png"`.
5. The `AircraftModelRepository` persists the new entity.
6. The system generates the full public URL for the image and returns an HTTP `201 Created` response with the HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | Validation fails (e.g., negative capacity) | HTTP `400 Bad Request` |
| 3 | The `modelName` already exists | HTTP `400 Bad Request` |
| 4 | File upload fails (I/O error) | HTTP `500 Internal Server Error` |

## 7. Design Justification
- **Physical Storage:** Storing images as physical files in `static/` significantly optimizes performance and simplifies access for the frontend.
- **Filename Sanitization:** Filenames are normalized based on the `modelName` to prevent path traversal attacks and ensure consistency.
- **HATEOAS Integration:** The returned URL enables the frontend to immediately display the image without requiring an additional search request.
