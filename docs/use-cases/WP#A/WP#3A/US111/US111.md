# US111 - Update Flight Route

## 1. User Story
> As a Backoffice Operator, I want to update the technical criteria of an existing flight route (e.g., estimated flight time, distance, minimum required range, and minimum capacity) to adapt to airspace or operational changes.

## 2. Acceptance Criteria
- The request must target an existing route via its unique `routeId`.
- The request payload can contain partial fields: `estimatedFlightTime`, `distance`, `minimumRange`, or `minimumCapacity`.
- Technical boundaries must be preserved during partial updates:
    - All numeric updates must be strictly positive values.
- On success, the system returns HTTP `200 OK` with the complete updated route payload and HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The targeted route must exist in the database.

## 4. Post-conditions
- The `Route` entity fields are updated in the database.
- Future flight schedules referencing this route will adapt to the new technical constraints.

## 5. Main Success Scenario
1. The actor sends a `PATCH /api/routes/{routeId}` request containing the updated fields.
2. The `RouteController` validates the syntax and boundaries of the fields present in the DTO.
3. The `UpdateRouteUseCase` retrieves the entity from the `RouteRepository`.
4. The entity applies the state modification safely through a domain business method.
5. The `RouteRepository` saves the changes (triggering optimistic locking tracking via `@Version`).
6. The system returns an HTTP `200 OK` response with the modified details.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user lacks valid credentials or roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 3 | The specified `routeId` does not match any record | HTTP `404 Not Found` (throws `RouteNotFoundException`) |
| 4 | Any provided value violates numeric criteria (e.g., distance <= 0) | HTTP `400 Bad Request` |

## 7. Design Justification
- **PATCH over PUT:** Chosen to allow partial resource modifications without requiring the client to resend immutable data such as origin/destination airport IATA codes.
- **Concurrent Integrity:** The entity tracks transactional updates through optimistic locking to prevent overlapping management conflicts.
