# US110 - Create Flight Route

## 1. User Story
> As a Backoffice Operator, I want to create a flight route between two registered airports, specifying technical criteria such as origin, destination, estimated flight time, distance, minimum required range, and minimum capacity.

## 2. Acceptance Criteria
- The request must provide `originIataCode`, `destinationIataCode`, `estimatedFlightTime`, `distance`, `minimumRange`, and `minimumCapacity`.
- Both `originIataCode` and `destinationIataCode` must correspond to active, registered airports.
- Distance and performance parameters must follow strict validation boundaries:
    - `distance` must be strictly positive (greater than 0).
    - `estimatedFlightTime` must be a strictly positive integer.
    - `minimumRange` and `minimumCapacity` must be strictly positive integers.
- The system must prevent creating a route where origin and destination are identical.
- On success, the system returns HTTP `201 Created` with the generated `routeId`, active status, and RESTful HATEOAS links (`self`, `history`).

## 3. Pre-conditions
- The actor must be authenticated and authorized with the `BACKOFFICE_OPERATOR` or `ADMIN` role (via Bearer JWT Token).
- The origin and destination airports must already exist and be operational in the system database.

## 4. Post-conditions
- The `Route` entity is successfully persisted in the database.
- The route status is initialized as `active = true`.
- The route is now available to be referenced in flight scheduling and template selection workflows (US212).

## 5. Main Success Scenario
1. The actor sends a `POST /api/routes` request with a valid JSON payload containing the route parameters.
2. The `RouteController` intercepts the request and validates the incoming DTO payload constraints.
3. The `CreateRouteUseCase` verifies with the `AirportRepository` that both IATA codes exist.
4. The `Route` domain entity is instantiated, executing self-validation checks (e.g., origin != destination).
5. The `RouteRepository` persists the new entity to the database, generating a unique `routeId` (UUID).
6. The system returns an HTTP `201 Created` response containing the route details along with `self` and `history` HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | Missing mandatory fields or invalid types (e.g., string instead of number) | HTTP `400 Bad Request` (intercepted by `GlobalExceptionHandler`) |
| 3 | Origin or destination airport does not exist in the database | HTTP `404 Not Found` (throws `AirportNotFoundException`) |
| 4 | Technical constraints violated (e.g., `distance <= 0` or origin matches destination) | HTTP `400 Bad Request` (throws `IllegalArgumentException` in domain/usecase) |

## 7. Design Justification
- **DTO Isolation:** Uses a clean request DTO to receive input, preventing HTTP/Web serialization logic from bleeding into the core domain layer.
- **Rich Domain Model:** The `Route` entity encapsulates its own business rules inside the constructor to guarantee that an invalid route can never exist in memory.
- **Use Case Decoupling:** The orchestration of checking existing airports and routing coordination is isolated inside `CreateRouteUseCase`, leaving the controller strictly responsible for HTTP protocol boundaries.
