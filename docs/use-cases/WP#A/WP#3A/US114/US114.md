# US114 - Retrieve Route Audit Change History

## 1. User Story
> As a Backoffice Operator, I want to view the audit change history log of a specific flight route to trace technical adjustments, revisions, and status changes over time.

## 2. Acceptance Criteria
- The request targets an existing route via its `routeId` through a sub-resource mapping `/history`.
- The response must return a chronological list of audit records detailing past parameter iterations.
- On success, returns HTTP `200 OK` containing the history timeline data model.

## 3. Pre-conditions
- The actor must be authenticated and authorized with administrative or operator privileges.
- The route resource track record must exist.

## 4. Post-conditions
- No state changes occur (Read-Only auditing operation).

## 5. Main Success Scenario
1. The actor sends a `GET /api/routes/{routeId}/history` request.
2. The `RouteController` identifies the resource.
3. The `GetRouteHistoryUseCase` contacts the auditing framework or repository.
4. The historical record logs are collected and mapped into a sequence DTO.
5. The system returns an HTTP `200 OK` collection response.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | Missing authentication tokens | HTTP `401 Unauthorized` |
| 3 | The `routeId` does not exist | HTTP `404 Not Found` |

## 7. Design Justification
- **Auditing Pattern:** Implemented to satisfy corporate tracking compliance. Isolating history logs under a secondary endpoint avoids inflating payload sizes during standard operational route lookups.
