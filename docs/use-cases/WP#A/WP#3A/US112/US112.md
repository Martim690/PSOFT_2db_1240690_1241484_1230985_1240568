# US112 - Toggle Route Operational Status

## 1. User Story
> As a Backoffice Operator, I want to change the operational status of a flight route (Deactivate/Reactivate) so that scheduling can be paused or resumed based on business demands.

## 2. Acceptance Criteria
- The request targets an existing route via its `routeId` using a dedicated status or toggle endpoint.
- If a route is deactivated, it remains in the system but cannot be used to schedule new flights.
- On success, the system returns HTTP `200 OK` showing the toggled `active` boolean status.

## 3. Pre-conditions
- The actor must be authenticated and authorized with the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The route must exist in the database.

## 4. Post-conditions
- The route's `active` state is permanently changed in the persistent database layer.

## 5. Main Success Scenario
1. The actor sends a `PATCH /api/routes/{routeId}/status` request with the intended status.
2. The `RouteController` maps the path identifier.
3. The `ToggleRouteStatusUseCase` loads the entity.
4. The entity changes its internal state (`active = false` or `active = true`).
5. The `RouteRepository` commits the updated record.
6. The system returns an HTTP `200 OK` status report.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is unauthenticated or lacks clearance | HTTP `401 Unauthorized` or `403 Forbidden` |
| 3 | The target `routeId` cannot be found | HTTP `404 Not Found` |

## 7. Design Justification
- **Soft Deletion Pattern:** Deactivation is used instead of a physical `DELETE` SQL command because existing scheduled flights require historical data consistency to satisfy route analytics constraints.
