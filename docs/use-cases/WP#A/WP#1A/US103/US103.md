# US 103 - Get Aircraft Details

## 1. Requirements Engineering

### 1.1. User Story Description

As a Backoffice Operator, I want to get the details of a specific aircraft instance using its registration number.

### 1.2. Customer Specifications and Clarifications

**From the specifications document:**
> "Operators must be able to consult the detailed profile of any registered aircraft using its unique tail number (registration number)."
> "The detailed view should present both the specific physical instance data (status, manufacturing date) and the general model specifications."

### 1.3. Acceptance Criteria

- **AC103-1:** The system must retrieve the exact aircraft matching the provided registration number.
- **AC103-2:** If the registration number does not exist in the database, the system must return a specific and clear error (e.g., `404 Not Found`).
- **AC103-3:** The response must return the full resource representation along with HATEOAS navigation links specific to that single aircraft (`self`, `update-status`, and `all-aircrafts`).

### 1.4. Found out Dependencies

- **Dependency on US102 (Register Aircraft Instance):** The system must have aircraft instances registered before they can be queried and retrieved.

### 1.5 Input and Output Data

**Input Data:**
- Typed data via URL Path Variable:
    - `registrationNumber` (String, e.g., `/api/aircrafts/CS-TPA`)

**Output Data:**
- (In)success of the operation (HTTP Status Codes: `200 OK` or `404 Not Found`).
- The JSON representation of the requested aircraft, including its embedded model data and RESTful `_links`.

## 3. Design - User Story Realization

### 3.1. Rationale

| Interaction ID | Question: Which class is responsible for... | Answer | Justification (with patterns) |
|:---------------|:--------------------------------------------|:-------|:------------------------------|
| Step 1 | ... interacting with the actor (client)? | `AircraftController` | **Controller (REST):** Receives the HTTP GET request and extracts the path variable. |
| | ... coordinating the US business flow? | `GetAircraftByRegistrationUseCase` | **Application Service / UseCase:** Orchestrates the retrieval flow, keeping controllers clean of data-access logic. |
| Step 2 | ... transferring the input data? | `@PathVariable String` | **Primitive:** Since the input is a single string (`registrationNumber`), a full DTO is unnecessary. |
| Step 3 | ... finding the aircraft in the database? | `AircraftRepository` | **Repository:** Executes the query (`findByRegistrationNumber`) abstracting the database mechanics. |
| Step 4 | ... handling the "Not Found" scenario? | `GetAircraftByRegistrationUseCase` | **Application Service:** Throws an exception if the `Optional` returned by the repository is empty, which is then translated to a 404 by the `GlobalExceptionHandler`. |
| Step 5 | ... formatting the successful response with links? | `AircraftController` | **Controller:** Uses the `toModel()` helper method to map the domain `Aircraft` entity into an `EntityModel<Aircraft>`, injecting the HATEOAS HAL standard response. |

### Systematization

According to the taken rationale, the conceptual classes promoted to software classes are:
- `Aircraft`
- `RegistrationNumber` (Value Object)

Other software classes (infrastructure and application layers) identified:
- `AircraftController`
- `GetAircraftByRegistrationUseCase`
- `AircraftRepository`

### 3.2. Sequence Diagram (SD)

![US103-SD](../../puml/SD/sd_us103.svg)

**Notice that:**
- The request requires no payload body, relying entirely on the URI path.
- The UseCase directly interrogates the Repository and returns the aggregate root.
- The Controller is responsible for transforming the pure domain object into an `EntityModel` (HATEOAS HAL format), keeping the UseCase completely unaware of HTTP/Web specifics.
