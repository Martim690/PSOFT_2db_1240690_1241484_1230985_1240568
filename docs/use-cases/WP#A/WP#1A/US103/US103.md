# US103 - Get Aircraft Details

## 1. User Story
> As a Backoffice Operator or ATCC, I want to view an aircraft's details given its registration number.

## 2. Acceptance Criteria
- The request must provide the `registrationNumber` as a path variable (e.g., `GET /api/aircrafts/CS-TPA`).
- The system must validate the format of the registration number before executing the query.
- On success, the system must return HTTP `200 OK` with the detailed aircraft payload (including model reference, status, seating capacity, and manufacturing date).
- If the requested `registrationNumber` does not exist in the database, the system must return HTTP `404 Not Found`.
- The response must include relevant RESTful HATEOAS links (e.g., `self`).

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The physical aircraft associated with the requested `registrationNumber` must have been previously registered in the system (via US102).

## 4. Post-conditions
- The system state remains entirely unchanged (this is a purely safe and idempotent read operation).
- The operator receives the requested aircraft information for visualization or further processing.

## 5. Main Success Scenario
1. The actor sends a `GET /api/aircrafts/{registrationNumber}` request.
2. The `AircraftController` intercepts the request and extracts the path variable.
3. The `GetAircraftByRegistrationUseCase` is invoked to fetch the aircraft.
4. The Use Case wraps the raw string inside a `RegistrationNumber` Value Object (validating its syntax).
5. The Use Case queries the `AircraftRepository` to find the exact match.
6. The `AircraftRepository` retrieves and returns the matching `Aircraft` aggregate root.
7. The system returns an HTTP `200 OK` response containing the serialized aircraft DTO and a `self` HATEOAS link.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 4 | The provided registration number violates the formatting regex rules (e.g., invalid characters) | HTTP `400 Bad Request` (thrown by `RegistrationNumber` Value Object constructor) |
| 5 | No aircraft is found for the given `registrationNumber` | HTTP `404 Not Found` (UseCase throws an exception, handled by `GlobalExceptionHandler`) |

## 7. Design Justification
- The raw URL parameter is immediately converted into a `RegistrationNumber` Value Object. This guarantees that malformed or malicious strings fail fast in memory and never reach the database query layer.
- This is a safe, side-effect-free `GET` operation. The API maps the internal `Aircraft` aggregate root to a dedicated DTO, strictly shielding the pure domain model from JSON serialization and web dependencies.
- Delegating the fetch logic to the `GetAircraftByRegistrationUseCase` ensures architectural consistency across the application. It leaves the Controller responsible only for HTTP mapping and HATEOAS link generation.
