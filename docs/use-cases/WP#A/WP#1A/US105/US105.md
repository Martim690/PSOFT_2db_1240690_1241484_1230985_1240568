# US105 - Update Aircraft Status

## 1. User Story
> As a Backoffice Operator (or Admin), I want to update the operational status of a specific aircraft instance so that the system accurately reflects its current availability (e.g., AVAILABLE, IN_FLIGHT, UNDER_MAINTENANCE).

## 2. Acceptance Criteria
- The request must use the `PATCH` method targeting the specific aircraft URI: `/api/aircrafts/{registrationNumber}/status`.
- The payload must contain the new `status`, which must strictly match a valid `AircraftStatus` enum value.
- If the requested aircraft does not exist, the system must return HTTP `404 Not Found`.
- The system must prevent concurrent updates (lost updates) using Optimistic Locking; conflicting updates should be safely rejected.
- On success, the system returns HTTP `200 OK` with the updated aircraft details and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The physical aircraft associated with the provided `registrationNumber` must already exist in the database.

## 4. Post-conditions
- The `Aircraft` aggregate root's `currentStatus` field is successfully updated.
- The database internal `@Version` of the aggregate is incremented to reflect the modification.

## 5. Main Success Scenario
1. The actor sends a `PATCH /api/aircrafts/{registrationNumber}/status` request containing the `UpdateAircraftStatusRequest` JSON payload.
2. The `AircraftController` catches the request and validates the syntax of the DTO.
3. The `UpdateAircraftStatusUseCase` queries the `AircraftRepository` to fetch the existing aircraft.
4. The Use Case invokes the explicit domain method (e.g., `updateStatus()`) directly on the `Aircraft` entity.
5. The `AircraftRepository` persists the updated entity, automatically verifying the `@Version` field to ensure no background modifications occurred.
6. The system returns an HTTP `200 OK` response containing the fully serialized, updated aircraft data alongside `self` and related HATEOAS links.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | The request payload contains an invalid, empty, or misspelled status | HTTP `400 Bad Request` (intercepted by Spring Validation) |
| 3 | No aircraft is found matching the given `registrationNumber` | HTTP `404 Not Found` |
| 5 | A concurrent update is detected by the database (JPA throws `ObjectOptimisticLockingFailureException`) | HTTP `409 Conflict` (or equivalent domain error handled by the `GlobalExceptionHandler`) |
