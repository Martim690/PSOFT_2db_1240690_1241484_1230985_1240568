# US 105 - Update Aircraft Status

## 1. Requirements Engineering

### 1.1. User Story Description

As a Backoffice Operator, I want to update the operational status of an aircraft.

### 1.2. Customer Specifications and Clarifications

**From the specifications document:**
> "The system must keep an accurate, real-time reflection of an aircraft's state (e.g., from AVAILABLE to IN_FLIGHT, or UNDER_MAINTENANCE)."
> "Because multiple operators might access and attempt to update the same aircraft profile simultaneously, the system must prevent concurrent modification conflicts (lost updates)."

### 1.3. Acceptance Criteria

- **AC105-1:** The system must allow changing the current status of an existing aircraft.
- **AC105-2:** The status must be validated against a predefined list of valid states (Enumeration).
- **AC105-3:** The system must handle concurrent updates safely. If two operators attempt to update the same aircraft simultaneously, the second save must fail and return a `409 Conflict` (Optimistic Locking).
- **AC105-4:** The API must use the `PATCH` HTTP method, as this is a partial update of the resource.
- **AC105-5:** The response must include HATEOAS links, explicitly providing a link to navigate back to the full aircraft details.

### 1.4. Found out Dependencies

- **Dependency on US102 (Register Aircraft Instance):** An aircraft instance must exist in the database before its status can be updated.

### 1.5 Input and Output Data

**Input Data:**
- Typed data via URL Path Variable:
    - `registrationNumber` (String)
- Typed data via JSON payload (DTO):
    - `currentStatus` (Enumeration: AircraftStatus)

**Output Data:**
- (In)success of the operation (HTTP Status Codes: `200 OK`, `400 Bad Request`, `404 Not Found`, or `409 Conflict`).
- The JSON representation of the updated aircraft, containing the new status, an incremented internal version number, and HATEOAS `_links`.

## 3. Design - User Story Realization

### 3.1. Rationale

| Interaction ID | Question: Which class is responsible for... | Answer | Justification (with patterns) |
|:---------------|:--------------------------------------------|:-------|:------------------------------|
| Step 1 | ... interacting with the actor (client)? | `AircraftController` | **Controller (REST):** Receives the HTTP `PATCH` request mapping the path variable and request body. |
| | ... coordinating the US business flow? | `UpdateAircraftStatusUseCase` | **Application Service / UseCase:** Orchestrates fetching the entity, invoking the domain mutation, and saving it. |
| Step 2 | ... transferring the new status data? | `UpdateAircraftStatusRequest` | **DTO:** Carries only the `currentStatus` field, shielding the domain from direct web exposure. |
| Step 3 | ... finding the aircraft in the database? | `AircraftRepository` | **Repository:** Queries the aggregate using the `registrationNumber`. |
| Step 4 | ... mutating the status safely? | `Aircraft` | **IE / Rich Domain:** The `Aircraft` entity exposes an `updateStatus()` method, ensuring state transitions happen inside the domain, not in the UseCase. |
| Step 5 | ... persisting the change and handling concurrency? | `AircraftRepository` | **Repository:** Executes the update. If the entity's `@Version` does not match the database version, it throws an `ObjectOptimisticLockingFailureException`. |
| Step 6 | ... translating the concurrency error to the client? | `GlobalExceptionHandler` | **AOP / Exception Handler:** Intercepts the locking exception and translates it into a clean HTTP 409 Conflict response. |
| Step 7 | ... formatting the successful response? | `AircraftController` | **Controller:** Wraps the updated entity in an `EntityModel`, providing a `self` link to the status endpoint, and an `aircraft-details` link back to the parent resource. |

### Systematization

According to the taken rationale, the conceptual classes promoted to software classes are:
- `Aircraft`
- `AircraftStatus` (Enumeration)

Other software classes (infrastructure and application layers) identified:
- `AircraftController`
- `UpdateAircraftStatusUseCase`
- `UpdateAircraftStatusRequest` (DTO)
- `AircraftRepository`
- `GlobalExceptionHandler`

### 3.2. Sequence Diagram (SD)

![US105-SD](../../puml/SD/sd_us105.svg)

**Notice that:**
- The UseCase pattern makes the transaction boundaries very clear (`@Transactional`).
- We specifically chose `PATCH` over `PUT`, since `PUT` implies replacing the entire resource, whereas we are only modifying a single property (`currentStatus`).
- The `GlobalExceptionHandler` completely removes the need to write `try-catch` blocks inside the Controller or UseCase, keeping the core logic highly readable.
