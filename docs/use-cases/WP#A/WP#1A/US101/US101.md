# US101 - Register Aircraft Model

## 1. User Story
> As a Backoffice Operator, I want to register an aircraft model with specifications including manufacturer, model name, seating capacity, fuel capacity, maximum range, and cruising speed.

## 2. Acceptance Criteria
- The request must provide `manufacturer`, `modelName`, `fuelCapacity`, `maximumRange`, and `cruisingSpeed`.
- `manufacturer` must be a predefined enum value (e.g., `AIRBUS`, `BOEING`, `BOMBARDIER`), not free text.
- `modelName` must be unique; attempting to register a duplicate model name must be rejected.
- Numeric constraints: `fuelCapacity`, `maximumRange`, and `cruisingSpeed` must be strictly positive integers (greater than 0).
- On success, the system must return HTTP `201 Created` with the model payload and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- An aircraft model with the specified `modelName` must not already exist in the database.

## 4. Post-conditions
- The `AircraftModel` is successfully saved to the database.
- The model is now ready to be referenced in aircraft registration processes and template selection workflows(US102).

## 5. Main Success Scenario
1. The actor sends a `POST /api/aircraft-models` request containing a JSON payload with the model's technical specifications.
2. The `AircraftModelController` validates the syntax and constraints of the incoming DTO.
3. The `RegisterAircraftModelUseCase` checks the `AircraftModelRepository` to ensure the `modelName` does not already exist.
4. The `AircraftModel` entity is instantiated, enforcing domain rules via its constructor.
5. The `AircraftModelRepository` persists the new entity to the database.
6. The system returns an HTTP `201 Created` response containing the saved model data and a `self` HATEOAS link.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | The request payload is missing mandatory fields or violates numeric constraints (e.g., negative fuel capacity) | HTTP `400 Bad Request` (intercepted by `GlobalExceptionHandler`) |
| 3 | The `modelName` already exists in the repository | HTTP `400 Bad Request` (throws `IllegalArgumentException` in UseCase) |

## 7. Design Justification
- `AircraftModel` is separated from physical aircraft to prevent duplicating technical specs across the fleet. It self-validates to ensure data integrity.
- Uses the `RegisterAircraftModelRequest` record to protect the core Domain entity from web and HTTP logic.
- The Use Case handles all business rules (like uniqueness checks), leaving the Controller strictly responsible for HTTP routing and responses.neration.
