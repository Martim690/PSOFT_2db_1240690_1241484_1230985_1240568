# US 104 - Search Aircrafts by Filters

## 1. Requirements Engineering

### 1.1. User Story Description

As a Backoffice Operator, I want to search and filter aircraft instances using specific criteria (e.g., status, model, manufacturing year).

### 1.2. Customer Specifications and Clarifications

**From the specifications document:**
> "Operators need to manage a large fleet and must be able to quickly filter aircraft based on their current operational status, the type of model, or their age (manufacturing year)."
> "The search mechanism should be flexible, allowing combinations of different filters."

### 1.3. Acceptance Criteria

- **AC104-1:** The system must allow searching by model name, operational status, and manufacturing year.
- **AC104-2:** All search parameters must be optional. If no parameters are provided, the system should return the complete list of aircraft.
- **AC104-3:** The response must return a `CollectionModel` (HATEOAS HAL format) containing the filtered list of aircraft.
- **AC104-4:** The root of the collection must contain a `self` link reflecting the exact query parameters applied, and an `all-aircrafts` link to allow the client to reset the search.

### 1.4. Found out Dependencies

- **Dependency on US101 & US102:** Searching for aircraft instances requires that models (US101) and physical instances (US102) are already populated in the system.

### 1.5 Input and Output Data

**Input Data:**
- Typed data via HTTP Query Parameters:
    - `model` (String, Optional)
    - `status` (Enumeration: AircraftStatus, Optional)
    - `year` (Integer, Optional)

**Output Data:**
- (In)success of the operation (HTTP Status Code: `200 OK`).
- A JSON HAL representation containing an `_embedded` list of aircraft that match the criteria. Each item contains its own specific links, and the root contains the collection links.

## 3. Design - User Story Realization

### 3.1. Rationale

| Interaction ID | Question: Which class is responsible for... | Answer | Justification (with patterns) |
|:---------------|:--------------------------------------------|:-------|:------------------------------|
| Step 1 | ... interacting with the actor (client)? | `AircraftController` | **Controller (REST):** Receives the HTTP GET request and extracts the query parameters using `@RequestParam(required = false)`. |
| | ... coordinating the US business flow? | `SearchAircraftsUseCase` | **Application Service / UseCase:** Orchestrates the retrieval flow, keeping controllers clean of data-access logic and dynamic query construction. |
| Step 2 | ... transferring the input data? | Primitive variables | **Primitives:** Since the inputs are optional URL query parameters, creating a strict DTO is unnecessary and over-engineers the solution. |
| Step 3 | ... dynamically finding the aircrafts? | `AircraftRepository` | **Repository:** Executes the dynamic queries abstracting the database mechanics (e.g., using Spring Data JPA custom queries or QueryDSL). |
| Step 4 | ... formatting the successful response with collection links? | `AircraftController` | **Controller:** Maps the list of domain `Aircraft` entities into a `CollectionModel<EntityModel<Aircraft>>`, injecting the HATEOAS standard responses (`self` with query params, and `all-aircrafts`). |

### Systematization

According to the taken rationale, the conceptual classes promoted to software classes are:
- `Aircraft`
- `AircraftStatus` (Enumeration)

Other software classes (infrastructure and application layers) identified:
- `AircraftController`
- `SearchAircraftsUseCase`
- `AircraftRepository`

### 3.2. Sequence Diagram (SD)

![US104-SD](../../puml/SD/sd_us104.svg)

**Notice that:**
- The request requires no payload body, relying entirely on the URI query parameters.
- The `AircraftController` handles the complexity of checking if parameters are null (falling back to a generic `listAll` flow if necessary) or forwarding them to the `SearchAircraftsUseCase`.
- The `CollectionModel` is built carefully to ensure the `self` link retains the exact search parameters the user submitted. This respects the highest maturity level of REST APIs (Richardson Maturity Model Level 3).
