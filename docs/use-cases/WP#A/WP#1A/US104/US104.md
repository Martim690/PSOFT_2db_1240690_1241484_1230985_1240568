# US104 - Search Aircrafts by Filters

## 1. User Story
> As an ATCC, I want to search for aircraft by model, status, or manufacturing year.

## 2. Acceptance Criteria
- The request must accept optional query parameters via URL: `model` (String), `status` (AircraftStatus Enum), and `year` (Integer).
- All search parameters must be optional and can be combined freely.
- If no parameters are provided, the system must seamlessly default to returning the complete catalog of aircraft instances.
- On success, the system must return HTTP `200 OK` with a `CollectionModel` (HATEOAS HAL format) containing the filtered list.
- The root of the JSON response must contain a `self` link that perfectly mirrors the exact query parameters applied, alongside an `all-aircrafts` link to allow clients to reset the search context.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- Aircraft models (US101) and aircraft instances (US102) should exist in the database for the search to return meaningful results.

## 4. Post-conditions
- The system state remains entirely unchanged (this is a purely safe, side-effect-free read operation).
- The operator receives a strictly filtered JSON array of aircraft matching their exact specifications.

## 5. Main Success Scenario
1. The actor sends a `GET /api/aircrafts/search` request with optional query parameters (e.g., `?status=AVAILABLE&year=2026`).
2. The `AircraftController` intercepts the request and extracts the primitive parameters using `@RequestParam`.
3. The Controller evaluates the input. If all parameters are null, it delegates to `ListAircraftsUseCase`. Otherwise, it invokes the `SearchAircraftsUseCase`.
4. The `SearchAircraftsUseCase` requests the filtered data from the `AircraftRepository`.
5. The `AircraftRepository` dynamically executes the matching database query and returns a list of `Aircraft` aggregates.
6. The Controller maps the resulting entities into a `CollectionModel<EntityModel<Aircraft>>`, carefully injecting the HATEOAS links.
7. The system returns an HTTP `200 OK` response.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 1 | The provided `status` parameter string does not match any valid `AircraftStatus` enum value | Spring Web automatically rejects the type conversion; returns HTTP `400 Bad Request` |
| 5 | The query executes successfully but no aircraft match the criteria | HTTP `200 OK` with an empty `_embedded` array (this is not considered an error state) |

## 7. Design Justification
- The `SearchAircraftsUseCase` delegates query construction entirely to the repository. This keeps the application service agnostic of database mechanics (SQL/JPA), adhering to the Dependency Inversion Principle.
- The API dynamically generates a `self` link that perfectly retains the combination of filters the client applied (e.g., `?status=AVAILABLE`), providing a fully navigable REST response.
- The Controller smartly evaluates the presence of query parameters, routing empty requests to a standard `ListAircraftsUseCase` and filtered requests to the `SearchAircraftsUseCase`, separating business concerns.
