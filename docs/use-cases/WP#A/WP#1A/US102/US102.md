# US 102 - Register Aircraft Instance

## 1. Requirements Engineering

### 1.1. User Story Description

As a Backoffice Operator, I want to register a new aircraft instance.

### 1.2. Customer Specifications and Clarifications

**From the specifications document:**
> "The system must keep track of individual physical aircraft."
> "Each physical aircraft must have a unique registration number (tail number), a manufacturing date, a specific seating capacity, and a reference to its technical specifications (Aircraft Model)."

### 1.3. Acceptance Criteria

- **AC102-1:** The aircraft's registration number (e.g., "CS-TPA") must be unique across the entire database.
- **AC102-2:** The aircraft must reference an already existing Aircraft Model (dependency).
- **AC102-3:** The seating capacity must be a strictly positive number.
- **AC102-4:** The response must return the created resource representation along with HATEOAS navigation links (including links to view its details, update its status, and return to the main list).

### 1.4. Found out Dependencies

- **Dependency on US101 (Register Aircraft Model):** A physical aircraft cannot be registered if its corresponding `AircraftModel` does not exist in the system's catalog.

### 1.5 Input and Output Data

**Input Data:**
- Typed data via JSON payload (DTO):
    - `registrationNumber` (String)
    - `modelName` (String - Reference to US101)
    - `manufacturingDate` (Date)
    - `seatingCapacity` (Integer)
- Selected data:
    - `currentStatus` (Enumeration: e.g., AVAILABLE, IN_FLIGHT, UNDER_MAINTENANCE)

**Output Data:**
- (In)success of the operation (HTTP Status Codes: `201 Created`, `400 Bad Request`, or `409 Conflict`).
- The JSON representation of the registered aircraft instance, including its embedded model data and RESTful `_links` (self, update-status, all-aircrafts).

## 3. Design - User Story Realization

### 3.1. Rationale

| Interaction ID | Question: Which class is responsible for... | Answer | Justification (with patterns) |
|:---------------|:--------------------------------------------|:-------|:------------------------------|
| Step 1 | ... interacting with the actor (client)? | `AircraftController` | **Controller (REST):** Receives the HTTP POST request and handles web-layer concerns. |
| | ... coordinating the US business flow? | `RegisterAircraftUseCase` | **Application Service / UseCase:** Orchestrates the flow, decoupling the business rules from the REST controller. |
| Step 2 | ... receiving and transferring the input data? | `RegisterAircraftRequest` | **DTO:** Carries the payload safely from the Controller to the UseCase. |
| Step 3 | ... verifying if the registration is unique? | `AircraftRepository` | **Repository:** Queries the database (`findByRegistrationNumber`) to enforce the uniqueness business rule (AC102-1). |
| Step 4 | ... fetching the required Aircraft Model? | `AircraftModelRepository` | **Repository:** Retrieves the aggregate root of the model based on the provided model name (AC102-2). |
| Step 5 | ... instantiating a new Aircraft? | `Aircraft` | **Creator:** The entity creates itself. The constructor applies domain validations (e.g., seating capacity > 0). |
| Step 6 | ... saving the created Aircraft? | `AircraftRepository` | **Repository:** Persists the new aggregate to the database. |
| Step 7 | ... formatting the successful response with links? | `AircraftController` | **Controller:** Maps the entity to an `EntityModel<Aircraft>` to automatically generate the HATEOAS standard response. |

### Systematization

According to the taken rationale, the conceptual classes promoted to software classes are:
- `Aircraft`
- `RegistrationNumber` (Value Object)
- `AircraftStatus` (Enumeration)

Other software classes (infrastructure and application layers) identified:
- `AircraftController`
- `RegisterAircraftUseCase`
- `RegisterAircraftRequest`
- `AircraftRepository`
- `AircraftModelRepository`

### 3.2. Sequence Diagram (SD)

![US102-SD](../../puml/SD/sd_us102.svg)

**Notice that:**
- The UseCase actively prevents data corruption by verifying both the uniqueness of the `RegistrationNumber` and the existence of the `AircraftModel` before allowing instantiation.
- The `Aircraft` entity is constructed atomically without the use of setters, maintaining a consistent state.
- Any domain violation triggers an `IllegalArgumentException`, which is intercepted and formatted by our `GlobalExceptionHandler` to ensure the client receives a clean `400 Bad Request`.
