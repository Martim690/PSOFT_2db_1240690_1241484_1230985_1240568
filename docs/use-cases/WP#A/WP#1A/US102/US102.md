# US102 - Register Aircraft Instance

## 1. User Story
> As a Backoffice Operator (or Admin), I want to register a physical, specific aircraft unit based on an existing catalog model so that the system can track individual aircraft instances and their operational statuses.

## 2. Acceptance Criteria
- The request must provide `registrationNumber`, `modelName`, `manufacturingDate`, `seatingCapacity`, and `currentStatus`.
- The `registrationNumber` must be globally unique across the database and follow specific formatting rules (e.g., uppercase alphanumeric with hyphens).
- The `modelName` provided must exactly match an already registered `AircraftModel`.
- Numeric constraints: `seatingCapacity` must be a strictly positive integer (greater than 0).
- On success, the system must return HTTP `201 Created` with the aircraft payload and RESTful HATEOAS links.

## 3. Pre-conditions
- The actor must be authenticated and authorized with either the `BACKOFFICE_OPERATOR` or `ADMIN` role.
- The `AircraftModel` referenced in the payload must have been previously registered (via US101).
- The provided `registrationNumber` must not already exist in the database.

## 4. Post-conditions
- A new `Aircraft` aggregate root is instantiated and persisted in the database.
- The new aircraft becomes immediately available for queries (US103/US104) and status updates (US105).

## 5. Main Success Scenario
1. The actor sends a `POST /api/aircrafts` request containing a JSON payload with the aircraft instance details.
2. The `AircraftController` validates the syntax and constraints of the incoming DTO.
3. The `RegisterAircraftUseCase` queries the `AircraftModelRepository` to ensure the requested model exists.
4. The `RegisterAircraftUseCase` queries the `AircraftRepository` to guarantee the `registrationNumber` is available.
5. The `Aircraft` entity is instantiated, enforcing domain rules via its constructor.
6. The `AircraftRepository` persists the new entity to the database (initializing its `@Version` to 0).
7. The system returns an HTTP `201 Created` response containing the saved aircraft data and a `self` HATEOAS link.

## 6. Alternative / Exception Flows
| Step | Condition | System Response |
|------|-----------|-----------------|
| 1 | The user is not authenticated or lacks the required roles | HTTP `401 Unauthorized` or `403 Forbidden` |
| 2 | The request payload is missing mandatory fields or violates constraints (e.g., negative seating capacity) | HTTP `400 Bad Request` (intercepted by `GlobalExceptionHandler`) |
| 3 | The referenced `modelName` does not exist in the repository | HTTP `400 Bad Request` (throws `IllegalArgumentException` in UseCase) |
| 4 | The `registrationNumber` already exists in the repository | HTTP `400 Bad Request` or `409 Conflict` |

## 7. Design Justification
- **Aggregate Separation:** The `Aircraft` is an independent Aggregate Root that holds a `@ManyToOne` reference to an `AircraftModel`. This prevents the duplication of technical specifications across thousands of physical planes.
- **Value Objects:** The `RegistrationNumber` is encapsulated as a Value Object. This guarantees that an invalid registration string can never be instantiated in memory, enforcing integrity before it even reaches the Entity.
- **Concurrency Protection:** The `Aircraft` entity utilizes a `@Version` field for Optimistic Locking. This is crucial to prevent race conditions when multiple operators attempt to update the status of the same physical aircraft later on.
- **Clean Architecture (Use Cases):** The `RegisterAircraftUseCase` orchestrates the validation of multiple repositories (`AircraftModelRepository` and `AircraftRepository`), keeping the Controller strictly focused on HTTP protocols.
